# ECGrid SOAP — .NET 10 Console (dotnet-svcutil Generated Proxy)

This project demonstrates how to call the ECGridOS SOAP API (v4.1) from a .NET 10
console application using a **typed proxy generated by `dotnet-svcutil`**. This approach
gives you IntelliSense, compile-time type safety, and automatic binding configuration —
at the cost of a code generation step whenever the WSDL changes.

---

## What It Demonstrates

- Generating a WCF-compatible typed proxy from the live ECGridOS WSDL
- Configuring `BasicHttpsBinding` programmatically (no `web.config`)
- Login → inbox poll → download → confirm → logout workflow using generated methods
- Closing/aborting the WCF channel correctly in a `finally` block
- Loading credentials from `IConfiguration` — never hardcoded

> The included `EcGridOSClientProxy.cs` is a **representative stub**.
> Replace it with the output of `dotnet-svcutil` before running the project.

---

## Generating the Real Proxy

Run these commands once (or whenever the WSDL changes):

```bash
# Install the tool globally (one-time)
dotnet tool install -g dotnet-svcutil

# Generate typed proxy in the current directory
dotnet-svcutil https://os.ecgrid.io/v4.1/prod/ECGridOS.asmx?WSDL \
    --outputDir . \
    --namespace *,ECGrid.Soap
```

This produces two files:
- `Reference.cs` — the generated client, interface, and all data types
- `dotnet-svcutil.params.json` — re-run parameters

After generation:
1. Delete `EcGridOSClientProxy.cs` (the stub file in this project)
2. Rename `Reference.cs` to something descriptive, e.g. `ECGridOSReference.cs`
3. `dotnet build` — `Program.cs` requires no changes

## Configuration

Open `appsettings.json`:

```json
{
  "ECGrid": {
    "UserName": "YOUR_USERNAME",
    "Password": "YOUR_PASSWORD"
  }
}
```

Or use environment variables:

```bash
export ECGrid__UserName="your-username"
export ECGrid__Password="your-password"
```

> Never commit real credentials to source control.

## How to Run

```bash
cd samples/soap/ECGrid-SOAP-dotnet10-Console-SvcUtil

# Only needed once — generate the real proxy (see above)
dotnet-svcutil https://os.ecgrid.io/v4.1/prod/ECGridOS.asmx?WSDL --outputDir . --namespace *,ECGrid.Soap

# Remove the stub after generation
# rm EcGridOSClientProxy.cs

dotnet run
```

Downloaded parcels are written to a `downloads/` subdirectory next to the executable.

## Comparing the Two SOAP Approaches

| | Manual HttpClient | dotnet-svcutil Proxy |
|---|---|---|
| Dependencies | `Microsoft.Extensions.Http` | `System.ServiceModel.Http` |
| Setup effort | None — just write code | Run svcutil once |
| Type safety | Manual XML parsing | Fully typed DTOs |
| IntelliSense | No | Yes |
| WSDL updates | Edit methods manually | Re-run svcutil |
| Binary size | Smaller | Larger |
| Best for | Minimal deps, few operations | Full API surface, larger teams |
